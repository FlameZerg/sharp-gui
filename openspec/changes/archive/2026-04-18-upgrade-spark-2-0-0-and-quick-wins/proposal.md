## Why

当前前端仍依赖 Spark 2.0 历史 Git 分支引用，锁文件实际解析到旧 commit（表现为 0.1.9），与官方已发布的 2.0.0 正式版产生版本漂移，带来可重复构建、问题排查和官方支持对齐上的风险。

同时，Spark 2.0 的 LoD/RAD/XR 相关能力已经在官方文档与示例中稳定化。现在迁移可以在低风险前提下获得明显收益（加载体验、性能稳定性、维护成本），且不需要先处理发布流程。

## What Changes

- 将前端 Spark 依赖从历史 Git 引用迁移到 npm 正式版 2.0.0，并重建锁文件确保解析源一致。
- 完成 API 与运行时行为对齐：清理历史遗留兼容用法，统一 SparkRenderer/SplatMesh 参数到正式版推荐路径。
- 引入“简单但收益明显”的新特性集成：
  - LoD 快速调优能力（面向桌面/移动/XR 的预设与参数控制）。
  - 非 LoD/LoD 快速切换能力（用于质量对比与问题定位）。
  - RAD 预构建 + paged 流式加载的可选接入路径（优先用于大模型场景）。
- 建立迁移验收与性能对比基线（功能可用性、渲染稳定性、XR 行为、加载和内存指标）。
- 明确范围外：本变更不包含发布流程、灰度发布策略与上线编排。

## Capabilities

### New Capabilities
- `spark-stable-upgrade`: Spark 依赖升级到正式版并确保安装来源、锁定结果与运行时版本一致。
- `spark-lod-quick-wins`: 提供低成本 LoD 质量/性能优化能力，包括可切换参数策略与对比手段。
- `spark-rad-streaming-workflow`: 支持 RAD 预构建与 paged 流式加载的工程化接入路径，并保留回退机制。

### Modified Capabilities
- 无

## Impact

- 依赖与构建：前端依赖声明、锁文件、安装与版本校验流程。
- 渲染与 XR：Spark 初始化参数、LoD/foveation 参数策略、XR 更新路径与兼容行为。
- 数据加载：模型加载流程新增 RAD/paged 可选分支及回退处理。
- 配置与文档：开发文档、迁移说明、验收清单与性能基线记录。
- 影响区域示例：frontend 依赖配置、viewer/xr 相关 hooks、状态管理与加载配置模块、OpenSpec 工件与项目规范文档。
