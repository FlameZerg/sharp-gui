## Context

当前侧边栏模型列表由 `App -> Sidebar -> GalleryList -> GalleryItem` 这一链路驱动：

- `App` 从 `useAppStore()` 直接读取较大范围的全局状态，并把 `galleryItems` 传给 `GalleryList`。
- `GalleryList` 直接对全部 `items` 执行 `map(...)` 渲染，没有列表虚拟化，也没有分页/分段挂载。
- `GalleryItem` 在每一行内再次订阅 `useAppStore()`，并使用浏览器原生 `loading="lazy"` 加载缩略图。
- `Sidebar` 的 `.content` 与 `GalleryList` 的 `.list` 都具备滚动语义，形成了不够清晰的滚动所有权。
- `useTaskQueue` 在任务完成后通过 `fetchGallery()` 全量刷新图库，再用 `setGalleryItems(...)` 替换整份列表。
- 后端 `/api/gallery` 每次请求都会扫描输出目录并拼装全部条目；当没有真实缩略图时，会把原图 URL 回填到 `thumb_url`。

这套结构在条目较少时可以工作，但在模型数增多后会叠加暴露几个问题：

- 列表 DOM 与图片节点线性增长，导致样式计算、布局、命中测试和图片解码成本同时上升。
- 无关的全局状态变化容易把列表和行组件一起拖入重渲染。
- 快速滚动时，原生懒加载与滚动容器的组合容易出现“缩略图短暂空白后再出现”的体验。
- 任务完成触发的整表替换会打断滚动上下文，并放大行组件 remount 与图片重建。
- 若列表项退回到原始大图作为缩略图，会把滚动性能和内存占用进一步拉低。

约束条件：

- 仅修改 React 前端与 Flask 后端，不触碰 `templates/index.html` 和 `ml-sharp/`。
- 保持现有 React + TypeScript + Zustand + CSS Modules 架构，不引入重量级 UI 框架。
- 需要同时兼顾桌面端与移动端侧边栏体验。
- 任何新增用户可见文本都必须维护 `frontend/src/i18n/en.json` 与 `frontend/src/i18n/zh.json`。

## Goals / Non-Goals

**Goals:**

- 让模型列表在大量条目下仍保持顺滑滚动与可预测的加载体验。
- 避免快速滚动时的缩略图空白闪烁、消失再出现和明显图片重建感。
- 保持用户当前的滚动上下文、选中项和可见内容，不因无关状态变化或任务完成刷新而被破坏。
- 为图库与缩略图建立清晰的前后端契约，避免列表行意外加载原始大图。
- 提供一个可回退、可验证的落地路径，而不是一次性高风险重写。

**Non-Goals:**

- 不优化 Three.js / Spark 模型渲染链路本身。
- 不新增搜索、筛选、分组、批量管理等新的产品能力。
- 不对侧边栏整体视觉风格做重设计。
- 不重构 Legacy 前端模板。

## Decisions

### 1) 将模型列表改为“单一滚动容器 + 虚拟化渲染”

**Decision:** 侧边栏将调整为稳定的结构分层：`header` 固定、`task queue` 保持自包含、`gallery list` 拥有独立滚动视口；模型列表仅渲染可视区及其 overscan 范围内的行。实现上引入一个薄适配层，默认使用 `@tanstack/react-virtual` 作为虚拟化引擎。

**Why:**

- 当前实现只是图片懒加载，不是列表级优化；即使图片不立即请求，所有行 DOM 仍然存在。
- 规范滚动所有权是虚拟化、预取和滚动位置保持的前提。
- 用成熟的虚拟化能力比手写窗口算法更稳，尤其是在 React 19、动态侧边栏宽度和移动端场景下。

**Alternatives considered:**

- 继续沿用全量 `map(...)` + `loading="lazy"`：无法解决 DOM 规模和图片节点生命周期问题。
- 手写 windowing / IntersectionObserver：可控但易踩边界问题，后续维护成本更高。
- 使用更老的 `react-window`：可行，但对动态容器变化和后续扩展性不如 `@tanstack/react-virtual` 灵活。

### 2) 收紧 gallery 相关状态订阅，避免无关更新拖慢列表

**Decision:** 不再让 `App`、`GalleryList`、`GalleryItem` 通过 `useAppStore()` 订阅整份大 store，而是改为 selector 化读取最小必要状态，并把行级展示所需的派生值在列表层预先计算后以下发 props 的方式传递。行组件本身应能在不相关状态变化时保持稳定。

**Why:**

- 当前 store 同时承载 viewer、loading、XR、quality、gallery 等多类状态，整 store 订阅会放大重渲染范围。
- 列表项数量一大，哪怕单行渲染不重，也会因为父组件/闭包重建而导致明显开销。
- 这一层优化能和虚拟化互补：虚拟化减少“同时存在的行数”，订阅隔离减少“每次更新时被动重算的行数”。

**Alternatives considered:**

- 只加 `React.memo`：能缓解一部分 props 变化，但无法解决整 store 订阅导致的父层重渲染压力。
- 把 gallery 单独拆成第二个 Zustand store：隔离更强，但会增加状态同步复杂度，当前阶段没必要引入第二个状态域。

### 3) 建立明确的缩略图契约，不再用原图顶替列表缩略图

**Decision:** 列表行只接受“真实缩略图资源”或“稳定占位状态”，不再把原始 `image_url` 作为列表缩略图兜底。后端图库接口只在真实 thumb 存在时返回 `thumb_url`；前端列表行维护明确的 `placeholder -> decoding -> ready -> error` 生命周期，并对可视区附近条目做提前加载。

**Why:**

- 当前 `thumb_url` 回退到原图会把几十 MB 的大图引入列表滚动路径，是性能与稳定性的大坑。
- 用户对列表缩略图的核心期待是“稳定感”，不是“尽可能快地显示任何图”。稳定占位比白块闪烁或大图抖动更可控。
- 明确区分“原图预览”和“列表缩略图”语义，也能让 `ImageViewer` / `/api/original/*` 与列表预览职责更清晰。

**Alternatives considered:**

- 继续依赖原生 `img loading="lazy"`：浏览器优化不可控，且无法避免空白闪烁。
- 缩略图缺失时继续直接回退原图：实现简单，但会持续制造大图解码与滚动卡顿。
- 全量预加载所有缩略图：会提前占用网络、内存和解码预算，不适合大图库。

### 4) 图库刷新改为“按 ID 协调更新”，保留滚动锚点与选中态

**Decision:** 任务完成后的图库刷新不再把 `galleryItems` 当作整份匿名数组直接替换，而是按稳定 `id` 做协调更新：保留未变化条目的身份，显式处理新增/删除/变更项，并在刷新后保持当前滚动锚点、选中模型和可见区上下文。

**Why:**

- 现在的“全量替换 + 重新渲染”会导致可见行 remount、图片重建和滚动体验断裂。
- 对用户来说，任务完成只是“多了一个模型”，不是“整个列表应该重置一遍”。
- 这一点对未来做分页、增量拉取或实时更新同样有利。

**Alternatives considered:**

- 继续整表替换，再尝试恢复 `scrollTop`：能修部分跳位，但不能解决可见行身份被销毁的问题。
- 上 WebSocket / SSE：可以做实时更新，但超出本次范围，也不是解决当前卡顿的必要条件。

### 5) 为缩略图增加专用服务语义与缓存控制

**Decision:** 后端将把“列表缩略图”作为明确资源对待，提供稳定的缩略图 URL 语义，并补齐缓存/条件请求支持；对于历史数据中的缺失缩略图，后端在可控范围内执行补修生成，避免前端长期落入异常路径。

**Why:**

- 现在 `/files/*` 是通用静态文件入口，缩略图与原图没有明确区分，也缺少针对列表场景的缓存策略。
- 丝滑滚动不仅取决于前端挂载策略，也取决于浏览器是否能重复利用已经拉取过的缩略图资源。
- 让后端主动补修缺失缩略图，比让前端长期承受异常分支更稳。

**Alternatives considered:**

- 完全复用当前 `/files/*` 路由而不调整语义：改动最少，但缓存策略和资源职责仍然模糊。
- 只靠前端占位，不修服务端缩略图：短期能止血，但会让历史数据一直存在性能尾部风险。

### 6) 保留一个短期回退路径，先验证再彻底切换

**Decision:** 在实现阶段保留旧列表实现或保留一个清晰的开关位，使虚拟化列表和新的缩略图契约可以在验证通过前快速回退。

**Why:**

- 列表虚拟化、滚动锚点保持和缩略图生命周期是多个改动面的组合，适合分阶段上线和回归。
- 当前模型浏览是主流程，出现回归时必须能够迅速退回稳定路径。

**Alternatives considered:**

- 一次性替换全部列表逻辑且不保留回退：开发期看似更干净，但风险高。

## Risks / Trade-offs

- [Risk] 虚拟化后键盘焦点、辅助功能或“滚动到当前选中项”逻辑出现回归  
  -> 缓解：统一通过列表适配层提供 `scrollToIndex` / `getItemElement` 能力，并在验收中覆盖键盘导航与选中项恢复路径。

- [Risk] 任务队列、侧边栏展开收起、移动端布局与新滚动容器相互影响  
  -> 缓解：先明确滚动所有权，再针对桌面/移动端分别做回归，避免继续保留双重滚动语义。

- [Risk] 历史数据缺失缩略图时，服务端补修导致 `GET /api/gallery` 响应变慢  
  -> 缓解：限制单次补修数量，优先上传时生成，必要时把历史补修放到后台任务或惰性修复逻辑中。

- [Risk] 缓存后的缩略图在文件被替换或删除后出现陈旧内容  
  -> 缓解：使用 `Last-Modified` / `ETag` 或基于文件变更时间的 version 参数，让浏览器缓存可正确失效。

- [Trade-off] 引入虚拟化依赖会增加一个小型前端库  
  -> 缓解：用本地适配层包住依赖，必要时可以在不重写业务组件的前提下替换实现。

- [Trade-off] 稳定占位策略会让个别图片在短时间内先显示 placeholder，而不是立即显示原图  
  -> 缓解：优先保证滚动稳定性与感知一致性，原图查看继续由 `ImageViewer` 承担。

## Migration Plan

1. 调整图库与缩略图契约：
   - 明确列表缩略图和原图的职责边界
   - 确保 `thumb_url` 只代表真实缩略图资源
   - 为缩略图请求补齐缓存/条件请求支持

2. 重构侧边栏列表结构：
   - 统一滚动所有权
   - 接入虚拟化列表适配层
   - 为列表行加入稳定的占位和预加载生命周期

3. 收紧 gallery 状态与刷新链路：
   - 将 `App` / `GalleryList` / `GalleryItem` 改为 selector 化订阅
   - 把任务完成后的图库刷新改成 merge-aware 协调更新
   - 保持选中项与滚动锚点稳定

4. 验证点：
   - 大量模型数据下的初始化、连续滚动和回滚滚动是否平滑
   - 快速滚动时缩略图是否稳定显示，无明显空白闪烁
   - 任务完成并插入新模型时，当前浏览上下文是否保留
   - 桌面端与移动端侧边栏收起/展开后列表状态是否正确
   - 缺失缩略图、缩略图加载失败、删除模型等异常路径是否稳定降级

5. 回退方案：
   - 在验证完成前保留旧列表渲染路径或显式开关位
   - 若新路径出现严重回归，可快速切回旧列表实现，同时保留服务端缩略图契约修复
