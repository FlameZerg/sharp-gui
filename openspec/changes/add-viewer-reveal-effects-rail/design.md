## 背景

Sharp GUI 当前的查看器叠加层被拆分在三个彼此独立的表面上：`Help` 锚定在查看器右上角，`QuickControls` 锚定在右下角并位于控制栏上方附近，而 `useViewer` 负责活跃的 Three.js/Spark 生命周期以及渲染循环。项目已经升级到 Spark 2.0.x，而 Spark 官方的 `splat-reveal-effects` 示例通过诸如 `dyno`、`SplatMesh.objectModifier` 和 `updateGenerator()` 之类的公共 API 演示了效果选择与重放。

该项目需求与官方演示在两个重要方面不同：

- 查看器渲染的是任意用户加载的 splat 模型，而不是特定于效果的示例资源。
- 新的效果切换器必须适配现有的玻璃拟态叠加层系统，并且在视觉上位于 Help 和 Quick Controls 之间，而不是放在底部控制栏或设置模态框内。

这意味着该设计在添加作为可选查看器侧增强功能的效果播放时，必须保留现有的模型加载、变换快捷控制、LoD 切换以及 XR 防护逻辑。

## 目标 / 非目标

**目标：**

- 添加一个专用的右侧 reveal-effects 侧栏，并与 Help 和 Quick Controls 共享垂直对齐。
- 将每个新的查看器会话默认设置为 `Magic` reveal effect。
- 允许用户在精选的 Spark reveal effects 之间切换，并在不重新加载模型资源的情况下重放当前动画。
- 保持新 UI 与项目的模糊玻璃拟态叠加层风格以及响应式叠加规则一致。
- 保留现有查看器在相机控制、模型变换、LoD 和模型加载生命周期方面的行为。

**非目标：**

- 不改动后端 API、任务队列或导出流水线。
- 不提供通用着色器编辑器、自定义效果创作 UI，也不持久化任意效果参数。
- 不尝试复现 Spark 示例中与资源绑定的相机编排或示例模型切换。
- 不修改旧版 `templates/index.html` 前端。

## 决策

### 决策：将 reveal-effect 选择保持在 `ViewerInstance` 本地

当前激活的 reveal effect 和 replay token 将存放在 `ViewerCanvas` / `ViewerInstance` 状态中，而不是放入全局 Zustand store。

原因：

- 当 `currentModelUrl` 变化时，查看器已经会重新挂载 `ViewerInstance`，这会自然地将每个新的查看器会话重置回 `Magic`。
- 这样可以让装饰性的播放状态不进入全局 store，避免新增持久化规则，并将耦合限制在查看器叠加层子树中。
- effect rail 和 `useViewer` 在同一个组件下是兄弟节点，因此可以显式传递状态，而无需引入更广泛的应用级依赖。

考虑过的替代方案：

- 使用 reveal-effect 状态扩展 `useAppStore`：可行，但比实际需要更重，并且会暗示需要持久化/恢复语义，而这不在范围内。
- 将效果存储在 localStorage 或 URL 状态中：否决，因为需求是会话级本地预览行为，而不是持久的用户偏好。

### 决策：在现有 `useViewer` 渲染循环内部集成效果播放

`useViewer` 将接收当前激活的 effect 和 replay token，为当前 `SplatMesh` 创建/更新 Spark `objectModifier`，并从已经驱动 controls 和 rendering 的同一个动画循环中推进 effect 时间。

原因：

- 当前 hook 已经负责 `SparkRenderer`、`SplatMesh` 以及渲染循环，因此它是与模型生命周期同步播放的最低摩擦位置。
- 复用现有动画循环可避免额外的定时器或渲染循环，并将 replay 重置、模型释放以及 XR 进入/退出保持在同一个生命周期边界内。
- Spark 官方示例使用的是公共效果钩子（`dyno`、`objectModifier`、`updateGenerator()`），这与项目对稳定 API 的要求一致。

考虑过的替代方案：

- 在每次切换效果时重新加载或替换模型资源：否决，因为应用必须把效果应用到当前选中的用户模型上，而不是 Spark 演示资源。
- 在 `useViewer` 外部运行一个单独的命令式动画控制器：否决，因为这会复制生命周期逻辑，并在模型或保真度设置重新初始化查看器时增加清理风险。
- 仅用 HTML/CSS 叠加层来模拟效果：否决，因为它无法对实际 splat 几何体进行动画处理。

### 决策：通过归一化的模型空间输入，将 Spark 的示例效果适配到任意用户模型

实现将保留官方精选效果集（`Magic`、`Spread`、`Unroll`、`Twister`、`Rain`），但会基于当前激活模型的局部包围体来适配效果输入，而不是使用 Spark 演示中针对示例资源的位置和缩放值。

原因：

- 用户上传的模型在缩放和原点上差异很大，因此官方示例中的硬编码位置在这个应用中并不可靠。
- 基于包围盒推导的归一化可以为效果公式提供稳定的半径/高度参考，并降低大模型或小模型动画过快、过慢或完全跑到屏幕外的概率。
- 这样可以使效果行为绑定到当前加载的模型，而不是绑定到一个精选演示场景。

考虑过的替代方案：

- 逐字复制 Spark 示例中的公式和常量：否决，因为这些值是为捆绑的演示资源调优的，在任意用户内容上会产生不一致的结果。
- 完全跳过归一化并接受按模型变化的差异：否决，因为默认的 `Magic` 需要在典型 gallery 模型上呈现出有意设计且可交付的效果。

### 决策：引入一个专用的 `ViewerRevealEffectsRail` 组件，并共享叠加层对齐 token

新的 rail 将作为一个独立的查看器组件，拥有自己的 CSS Module，同时会调整 `Help` 和 `QuickControls`，使它们对齐到共享的右侧叠加层基线。

原因：

- 独立组件符合项目的组件约定，并且可以让特定于效果的 UI 与帮助内容和变换控制隔离开。
- 共享对齐值可以减少三个独立叠加层之间的漂移，并让“同一条垂直线”这一要求更易于维护。
- 复用现有玻璃 token（`--glass-bg`、`--glass-border`、`--glass-blur`、`--glass-shadow`）可以保留既有的 Apple 风格叠加层视觉语言。

考虑过的替代方案：

- 将效果切换放进 `QuickControls`：否决，因为需求是一个始终可用的侧边 rail，而现有 quick-controls 的能力有意限定在变换/交互设置范围内。
- 将效果按钮放进底部 `ControlsBar`：否决，因为这会破坏所要求的右侧放置方式，并且会与 reset/fullscreen/share 等操作竞争空间。
- 在每个叠加层中独立硬编码 `right: 16px` 偏移：否决，因为随着布局演进，这种对齐方式仍然很脆弱。

### 决策：在叠加层受限时完全隐藏 rail，同时保持基础预览行为不受影响

该 rail 将遵循与其他查看器叠加层相同的可用性规则：当没有可用的活动预览时，它应消失；并且当查看器进入 XR 呈现时，它应被完全隐藏，而不是显示为禁用态。

原因：

- 该需求是查看器增强功能，而不是渲染的硬依赖。
- XR 已经改变了查看器交互假设；在这种情况下继续保持 rail 激活只会增加复杂度，而没有明确的用户价值。
- 安全降级即使在效果播放不可用时，也能保留 Help 和 Quick Controls 的行为。

考虑过的替代方案：

- 在 XR 中保持 rail 完全可交互：否决，因为在 XR 会话期间更难保证指针可达性和视觉密度。
- 在 XR 中显示禁用占位 rail：否决，因为这会增加视觉噪音，但不提供实际可操作价值。
- 在效果 UI 可渲染之前阻塞查看器：否决，因为效果切换必须保持为可选功能。

### 决策：移动端采用“紧凑入口 + 临时展开面板”，不常驻完整竖向 rail

移动端不直接复用桌面端的完整竖向 rail，而是默认收敛为一个紧凑的浮动入口。用户点击入口后，再展示一个临时的效果面板；完成切换后面板收起，只保留入口。

原因：

- 手机屏幕上的可用预览区域更小，长期悬浮的完整竖向 rail 会更明显地切走模型观察空间。
- reveal effect 属于低频切换、短时使用的增强功能，不值得长期占用一整列画面。
- 移动端更需要更大的触控命中面积，而不是细窄的常驻 rail；临时展开面板更容易保证 44px 以上的触控目标。
- 为减少与查看器旋转、缩放、平移以及系统边缘手势的冲突，移动端不应依赖额外的滑动手势来完成效果切换。

考虑过的替代方案：

- 在移动端保留完整竖向 rail 但缩小尺寸：否决，因为即使缩小也会持续侵占画面，同时点击目标会变差。
- 改成常驻的顶部或底部横向 tabs：否决，因为会与现有 Help、ControlsBar 以及系统手势更容易发生竞争。
- 使用滑动手势切换效果：否决，因为会与查看器已有手势冲突，误触成本更高。

## 风险 / 权衡

- [Spark 示例效果在异常模型缩放下可能仍需调优] -> 在可用时从包围盒推导归一化输入，使用几个有代表性的 gallery 资源进行验证，并将常量集中管理以便快速调整。
- [逐帧 modifier 更新可能增加渲染开销] -> 复用现有动画循环，避免额外定时器，并且仅在效果播放激活且 mesh 存在时触发 mesh generator 更新。
- [小屏幕上的右侧叠加层拥挤] -> 移动端默认只保留紧凑入口，点击后使用临时展开面板承载效果选择，完成后自动或显式收起。
- [查看器重新初始化路径可能意外丢失效果状态] -> 将效果状态保留在 `ViewerInstance` 本地，在重新挂载时有意重置它，并在高保真切换和模型重载后加入 replay/切换验证。

## 迁移计划

1. 为 reveal-effects rail 添加一个新的查看器叠加组件，并增加 effect 标签、工具提示以及 replay affordance 的本地化字符串。
2. 使用 reveal-effect 输入扩展 `useViewer`，并将 Spark modifier / replay 处理绑定到当前 mesh 生命周期。
3. 从当前激活模型的 bounds 对 effect 输入进行归一化，并验证其与 transform controls、LoD 设置和相机重置的共存。
4. 调整 Help 和 Quick Controls 的位置，使其与新的 rail 共享一个通用的右侧对齐基线。
5. 在发布前验证桌面端、移动端紧凑入口、无模型以及 XR 状态。

回滚策略：

- 从 `ViewerCanvas` 中移除 rail 组件，并在 `useViewer` 中绕过 reveal-effect modifier 设置，这样应用就会在不影响后端的情况下恢复到当前的查看器行为。

验证点：

- 打开模型时默认从 `Magic` 开始。
- 切换效果会改变当前动画，而无需离开查看器。
- Replay 会重启当前激活的效果，而不会重新获取模型。
- 在桌面端，Help、rail 和 Quick Controls 保持对齐且可操作。
- 在移动端，仅显示紧凑入口，展开后不会长期遮挡主渲染画面。
- 在 XR 模式下，rail 完全隐藏。

## 开放问题

- 移动端临时展开面板最终采用“右侧锚点弹出”还是“贴底部的浮动小面板”，需要在真实设备上确认哪一种对主渲染画面的遮挡更少。
- 在真实用户模型上进行视觉 QA 后，某些 Spark 示例常量可能需要轻微的逐效果调优；这属于预期中的实现校准，而不是设计阻塞项。
