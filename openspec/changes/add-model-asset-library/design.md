## Context

Sharp GUI 当前模型入口仍以侧栏模型列表为主，后端主要从 `outputs/*.ply` 扫描生成结果，再通过同名 `.spz`、输入图片、视频源帧和 sidecar 元数据补齐展示信息。这个模型适合少量生成结果，但当模型数量增长、来源扩展到外部 PLY/SPZ/SPLAT/RAD 文件后，浏览、筛选、对比和回看效率会明显落后于已经具备瀑布流、分页、排序、导入能力的本地照片图库。

本变更把“模型列表”升级为“模型资产库”：生成结果和导入模型进入同一个资产索引，主模型页从单调列表改为接近用户确认示例图的信息架构和工作台布局，并在项目现有 Apple 毛玻璃设计语言下提供顶部工具栏、左侧导航/任务/近期模型、右侧选中详情面板、模型卡片封面、格式徽标、筛选排序、导入和打开查看能力。

必须遵守项目现有约束：前端使用 React + TypeScript + Zustand + i18n，样式使用 CSS Modules 与 CSS Variables；后端是 Flask 模块化服务，继续使用文件系统持久化，不引入数据库或 ORM；不修改 `ml-sharp/`、Legacy `templates/index.html` 和 `static/lib/`。

## Goals / Non-Goals

**Goals:**

- 将模型入口主区域升级为高保真模型资产库 UI，尽可能还原示例图的信息架构、顶部筛选工具栏、网格卡片、右侧详情面板和左侧辅助导航，同时遵循项目现有 Apple 毛玻璃视觉规范与深浅色模式适配。
- 支持生成模型与导入模型统一浏览，覆盖 PLY、SPZ、SPLAT、RAD 等当前 viewer 已知或计划支持的模型格式。
- 支持本机文件选择、拖拽、批量上传，以及已授权局域网设备上传模型到受控资产库。
- 为模型资产提供稳定封面：生成模型优先复用源图片或视频帧，外部导入模型优先用浏览器端离屏渲染生成封面，失败时使用格式感知占位图。
- 支持用户手动编辑模型资产的显示名称、标签、备注和封面，让外部导入模型可以被整理成更易浏览的资产。
- 保留现有生成、查看、下载、导出、删除和旧 `/api/gallery` 兼容路径，避免一次性破坏既有用户工作流。
- 在大模型集合下保持浏览响应：分页或游标加载、稳定缩略图状态、选择状态保留、避免全量同步渲染。
- 将新增导入和封面写入端点纳入现有局域网访问控制与远程生成权限矩阵。

**Non-Goals:**

- 不修改 `ml-sharp/` 推理引擎，也不改变模型生成算法或训练流程。
- 不重写 Legacy `templates/index.html`，本次仅面向主 React 前端。
- 不引入数据库、ORM、Tailwind、CSS-in-JS、Sass 或 axios。
- 不在后端引入重型 3D 渲染服务；首版封面渲染由浏览器端完成，后端只负责接收、校验和缓存缩略图。
- 不实现外部目录“引用型模型相册”或持续监控 NAS 目录；首版导入会复制文件到 workspace 下受控资产目录。
- 不实现模型跨相册、集合、文件夹或多库管理；首版所有模型统一放在同一个模型资产库中，通过来源、格式、时间等筛选提高浏览效率。
- 不把远程已授权用户提升为 owner；远程导入只在明确允许远程生成/导入时开放，owner-only 管理动作仍保持本机限制。

## Decisions

### Decision: 使用文件系统模型资产索引作为兼容层

模型资产库将新增一个 workspace 内的模型资产索引，统一描述生成模型与导入模型。索引记录稳定 ID、显示名、来源类型、主文件、可选关联文件、格式、大小、时间、缩略图、用户编辑字段、元数据摘要和导入状态；现有 `outputs/*.ply/.spz` 通过兼容扫描或迁移补充到索引中。

Why: 现有后端已经是无数据库架构，照片图库也以缓存索引和文件系统路径为核心。文件系统索引能保留便携版部署特性，降低迁移风险，并让旧生成结果不用立即搬迁。

Alternatives considered:

- 引入 SQLite 或 ORM：查询能力更强，但违反当前项目无数据库约束，也会增加 Windows 便携包迁移和备份复杂度。
- 每次请求都扫描 `outputs/` 与导入目录：实现简单，但在大量模型、NAS 或慢盘下会拖慢启动和刷新，不适合作为主浏览路径。
- 只扩展现有 `/api/gallery` 返回结构：短期改动少，但会把导入、分页、封面、筛选和兼容逻辑挤进旧模型列表语义，后续难以维护。

### Decision: 资产资料允许用户手动编辑，但不改变源文件身份

模型资产支持用户手动编辑显示名称、标签、备注和封面。编辑内容写入资产索引或 sidecar 元数据，不重命名模型源文件，不改变资产稳定 ID，也不影响 viewer 使用的真实模型文件路径。

Why: 外部导入模型往往来自不同工具或设备，文件名可能不可读，系统自动派生的封面也可能不够理想。允许用户整理名称、标签、备注和封面，可以显著提升大量模型下的回看效率，同时保持底层文件管理简单可靠。

Alternatives considered:

- 只使用文件名和系统派生元数据：实现最简单，但用户很难把外部模型整理成可识别资产。
- 允许重命名源文件：看起来更直观，但容易破坏关联格式、下载链接、旧索引和 viewer 缓存。
- 首版就支持复杂标签体系、集合和批量编辑：整理能力更强，但会扩大范围；本次先支持单个资产的基础编辑。

### Decision: 导入模型复制到受控资产目录

首版导入会将本机或局域网上传的模型文件复制到 workspace 下受控模型资产目录，并为每个导入批次创建资产记录。文件名、扩展名、大小和路径都由后端校验，返回给前端的是资产 ID 与受控 URL，而不是用户原始绝对路径。

Why: 复制导入能保证 `/files/*` 静态服务 allowlist、删除、下载、缩略图和权限控制都在 Sharp GUI 管理范围内，也避免把任意本机/NAS 路径暴露给远程客户端。

Alternatives considered:

- 引用外部路径不复制：节省磁盘空间，但权限、安全、跨平台路径和离线可用性复杂，尤其局域网客户端不应知道主机绝对路径。
- 使用浏览器本地 blob URL 只临时预览：已有拖拽行为接近这个方案，但模型不会进入资产库，刷新后丢失，不满足统一管理诉求。
- 为模型做类似照片图库的外部目录相册：适合未来扩展，但首版范围更大，需要目录扫描、索引失效、路径权限和 NAS 性能策略。

### Decision: 新增模型资产 API，同时保留旧 gallery 兼容

后端新增 `/api/model-assets` 系列端点用于分页列表、详情、导入、资产资料编辑、缩略图写入/刷新、删除、下载和打开查看所需 URL。旧 `/api/gallery` 在过渡期继续返回现有结构，或由模型资产服务适配生成原有字段。

Why: 新 API 可以表达资产来源、格式、分页、筛选、排序、封面状态和导入状态；兼容旧 API 可以降低前端分阶段迁移和外部脚本破坏风险。

Alternatives considered:

- 直接替换 `/api/gallery`：端点少，但会把旧调用方一次性暴露在破坏性变更中。
- 完全保留旧列表，前端自行合并导入结果：会让浏览逻辑分裂，选择、删除、下载和封面状态难以一致。

### Decision: 主模型页改为资产库工作台，侧栏降级为辅助导航

React 主模型视图将引入 `ModelAssetLibraryView` 一类的工作台组件：左侧为生成/导入入口、模型/图库一级切换、任务队列、近期模型和存储摘要；中间为标题、筛选/排序/密度/导入/选择/打开工具栏与模型卡片网格；右侧为选中资产的封面、动作和详情/元数据标签页。侧栏列表不再作为大量模型的主要浏览路径。

Why: 用户明确认可示例图，并希望尽可能还原。该结构能把浏览、筛选、详情和动作放在同一屏内，信息密度高于单列列表，也和照片图库的卡片式浏览形成一致体验。

Alternatives considered:

- 只给现有侧栏列表增加搜索和分组：改动小，但仍受侧栏宽度限制，无法提供足够封面和详情空间。
- 复用照片图库瀑布流原样展示模型：可以减少组件量，但模型资产需要格式徽标、打开查看、导入状态、右侧 3D 元数据等不同交互。
- 做营销式首页或大 hero：不适合工具主入口，且会降低高频管理效率。

### Decision: 封面生成采用“源媒体优先、浏览器离屏渲染、稳定占位”三级策略

生成模型优先使用输入图片、视频源帧或已有缩略图作为封面。导入模型在前端加载模型后，使用 Three.js/Spark 在隐藏画布中进行有界离屏渲染，生成小尺寸 PNG/WebP 后提交到后端缓存。若模型过大、格式暂不支持、加载失败或浏览器资源不足，则使用带格式、颜色和状态的稳定占位图。

Why: 后端当前没有通用 3D 渲染管线，浏览器端已经具备 Spark/Three 依赖。离屏渲染可以利用现有 viewer 能力，避免在 Python 后端新增重型图形依赖，同时让导入 SPZ/SPLAT/RAD 获得可识别封面。

Alternatives considered:

- 后端用 OpenGL/软件渲染生成封面：可以集中处理，但依赖重、跨平台难，Windows 便携包和无 GPU 主机风险高。
- 上传时强制用户提供图片封面：质量可控，但增加导入摩擦，也不适合批量模型。
- 永远使用格式占位图：实现最稳，但浏览效率提升有限，不符合“图库式浏览”的核心目标。

### Decision: RAD 首版作为可打开格式展示，不新增流式状态面板

RAD 首版在资产库中按一种模型格式展示：卡片和详情面板显示 RAD 徽标、文件大小、修改时间和默认打开能力；viewer 仍可沿用已有 RAD 分页加载能力。资产库首版不额外展示 RAD 内部的流式加载状态、分页 chunk 数量、LoD 层级、预取进度或 RAD 生成参数。

这里的“RAD 流式状态 / LoD 元数据”指的是 RAD 大场景可能按层级和分页块加载：先显示粗略版本，再逐步加载高细节块。如果把这些信息放进资产库，就需要记录和展示“这个模型有几个层级、多少页、加载到哪一页、当前 viewer 是否正在流式加载”等高级诊断信息。它对调试大场景有用，但对首版图库式浏览不是必需。

Why: 用户当前目标是提升模型浏览、导入、预览和整理效率。把 RAD 先作为可打开格式纳入统一资产库，可以覆盖首版主要价值；专属流式诊断会牵涉 viewer 运行时状态、RAD 生成流程和更多元数据解析，容易扩大范围。

Alternatives considered:

- 首版展示 RAD 流式进度和 LoD 结构：对高级用户有价值，但需要 viewer 与资产库共享运行时状态，实施成本和测试范围更大。
- 完全不支持 RAD 资产：实现更简单，但和现有 RAD viewer 能力及用户导入外部模型的目标不一致。
- 仅在详情面板预留 RAD 高级字段但不填充：会制造空洞 UI；首版保持清晰的格式展示，未来需要时再扩展。

### Decision: 新增端点遵循现有局域网权限模型

模型资产读取在局域网访问控制启用时要求 owner 或已认证会话；模型导入、资产资料编辑和封面写入属于会改变 workspace 的操作，远程客户端只有在访问控制启用且 owner 明确开启现有“允许远程生成”开关时才可执行；删除、配置和全局批处理等 owner-only 行为继续限制为本机 owner。

Why: 导入模型、编辑资料和写入封面会占用磁盘并改变私有数据，风险接近远程生成。复用现有“允许远程生成”开关可以减少设置复杂度，并让“局域网设备是否能写入 workspace”保持一个清晰入口。

Alternatives considered:

- 远程认证后默认允许导入：更方便，但会让任何拿到访问码的人写入大量文件。
- 导入只允许 localhost：最安全，但不满足用户提出的局域网设备导入诉求。
- 新增独立“远程导入”开关：权限更细，但用户已确认首版复用“允许远程生成”即可，避免增加设置复杂度。

### Decision: UI 高保真还原通过组件化布局和视觉验收保证

实现时将把示例图拆成稳定布局区域和可复用控件：侧栏、顶部栏、模型卡片、详情面板、状态徽标、密度控制、选择模式、分页区域和空态。样式使用 CSS Modules + CSS Variables，并复用 `.agents/rules/styling-guide.md` 中定义的 Apple 风格玻璃态、系统字体、设计 token、轻量过渡、触控适配和 `prefers-color-scheme` 深浅色策略；关键桌面和移动宽度通过 Playwright 或手工截图验证，确保文本不溢出、控件不重叠、卡片尺寸稳定、层级关系接近示例。

视觉验收基线：

- 桌面宽屏优先还原示例图的三栏工作台：左侧固定辅助栏，中间为可伸缩模型网格，右侧为固定宽度详情面板；三栏都应占据首屏高度，并通过项目既有玻璃态面板、边框、阴影和文本层级形成清晰的信息层次。
- 顶部区域应保持示例图的信息顺序：标题和总数在左上，来源筛选分段控件靠左，排序、密度、刷新、导入、选择和打开操作靠右；按钮应使用图标或图标加短文本，保持紧凑工具型界面。
- 模型卡片应保持固定比例和稳定信息层级：上方大封面，下方格式徽标、名称、大小、来源和时间；选中卡片使用项目强调色 token 驱动的描边与勾选状态，悬浮或选中时显示查看、下载、导出、删除等快捷操作且不引起布局跳动。
- 右侧详情面板应接近示例图结构：顶部大预览，中部标题与编辑入口，主要操作按查看、下载、导出、分享（如现有分享能力可用或以禁用态占位）、删除排列，下方使用 Details / Metadata 标签承载字段。
- 底部浏览区应提供接近示例图的分页或批次导航体验，包括当前页、相邻页、前后翻页和每页数量选择；若后端采用游标式加载，也应保留清晰的加载边界和批次大小控制。
- 视觉语言不得从示例图硬拷贝为固定深色主题；必须以项目现有高级 Apple 风格为准，复用 CSS Variables、玻璃态 token、系统字体、项目强调色和 Light-first + Dark override 策略。真实模型缩略图应成为卡片视觉主角；背景氛围、纹理或装饰只能沿用或轻量扩展现有设计系统，不得遮挡内容或变成装饰主角。
- 允许因真实数据、语言长度、窗口宽度和现有设计系统 token 产生小幅差异；不允许偏离为营销页、普通表格页、单列列表页或照片图库原样套壳。

Why: “尽可能 100% 还原”需要在规格中明确布局和验收，而不是只描述“更好看”。组件化也能让后续功能继续扩展而不牺牲一致性。

Alternatives considered:

- 单个大组件快速实现：还原快，但状态、响应式和测试会失控。
- 先做纯功能再美化：风险是实现完成后视觉偏离用户确认的方向，返工成本更高。

## Risks / Trade-offs

- [现有生成结果缺少完整资产元数据] -> 首次进入资产库时从文件名、sidecar、输入图片和文件 stat 派生最小元数据；缺失字段在 UI 中显示为未知或占位，不阻塞浏览。
- [导入大模型或批量模型导致磁盘压力] -> 后端限制单文件大小、总批次大小和支持扩展名，UI 显示存储摘要，并把失败项逐条返回。
- [浏览器端封面渲染可能失败或耗时] -> 封面生成排队并发受限，只针对可见或新导入资产触发；失败后缓存失败状态并显示稳定占位，避免反复卡顿。
- [资产索引与文件系统不同步] -> 列表读取时验证关键文件存在，缺失资产显示不可用状态；显式刷新可重建索引摘要；删除操作以资产 ID 和受控路径为准。
- [局域网导入增加安全面] -> 所有写入端点复用访问控制、扩展名 allowlist、路径归一化、文件名净化和受控目录；远程写入跟随现有“允许远程生成”开关，默认不开放。
- [高保真 UI 可能牵涉较多组件改造] -> 先以主模型页工作台为边界，保留 viewer 与生成流程接口；用分阶段任务减少一次性重写。
- [旧 `/api/gallery` 与新资产 API 语义重复] -> 旧端点只作为兼容层保留，前端新视图统一消费新资产 API，后续归档时再评估是否废弃旧字段。

## Migration Plan

1. 新增模型资产目录、缩略图缓存目录和索引文件，不移动或删除现有 `outputs/`、`inputs/` 和已有缩略图。
2. 新增模型资产服务，先以只读兼容方式把现有 `outputs/*.ply/.spz` 映射为资产，并保证旧 `/api/gallery` 响应不退化。
3. 新增导入、详情、缩略图写入和下载端点，接入访问控制与路径 allowlist。
4. 前端新增模型资产 API、类型、store 状态和主资产库组件，保留现有 viewer 打开模型的加载路径。
5. 将模型入口切换到资产库工作台；旧侧栏模型列表保留为近期模型或辅助入口，直到新视图验证稳定。
6. 添加中英文 i18n 文本、后端权限/路径/导入测试、前端关键状态测试，并进行桌面与移动截图验收。

Rollback strategy:

- 如果新资产库 UI 或 API 出现阻断问题，可以将前端模型入口临时切回旧 gallery 消费路径；由于首版不移动现有生成文件，旧列表仍可工作。
- 可以保留新索引和导入目录但停止使用新端点；旧 `outputs/` 生成结果不受影响。
- 若某个导入资产导致索引解析失败，后端应跳过该资产并记录错误，不影响旧模型浏览。

Validation points:

- 旧生成结果在新资产库中可见，并可继续打开 viewer、下载和删除。
- 本机批量导入 PLY/SPZ/SPLAT/RAD 后，资产出现在网格中，刷新页面后仍存在。
- 已授权且开启“允许远程生成”的局域网客户端可以上传模型、编辑资料和写入封面；未认证或未启用远程生成权限的客户端被拒绝。
- 导入模型封面成功时显示渲染图，失败时显示稳定占位，列表滚动不卡顿。
- 200+ 模型资产下，筛选、排序、滚动、选择和详情面板保持响应。
- 桌面宽屏布局按视觉验收基线对照示例图检查，覆盖三栏比例、工具栏顺序、卡片结构、详情面板、分页区域和项目 Apple 毛玻璃层级；浅色和深色系统主题下均需保持可读、协调且无布局破坏；移动/窄屏无文本溢出、控件重叠或横向滚动。

## Open Questions

暂无未决问题。已根据用户反馈确认以下方向：

- RAD 首版仅作为可打开格式展示，不新增资产库层面的流式状态、分页 chunk、LoD 层级或加载诊断面板；viewer 仍可沿用已有 RAD 加载能力。
