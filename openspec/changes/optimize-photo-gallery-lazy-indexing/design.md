## Context

Sharp GUI 的本地图库已经支持配置本机/NAS 相册、图片和视频混合浏览、缩略图/poster、预览、下载、筛选、排序和照片转 3D。当前实现有 `.photo-gallery-cache/index.json` 作为轻量索引，但普通请求并没有真正走缓存优先路径：

- 应用启动时 `App.loadPrivateData()` 会串行等待 `fetchPhotoAlbums()`，因此用户只想进入模型预览页时也会被图库相册扫描阻塞。
- `/api/photo-albums` 对每个配置相册调用 `build_photo_album_response()`，该函数直接进入 `scan_photo_album()`。
- `/api/photo-albums/<album_id>/photos` 每次分页、筛选、排序请求都会先调用 `scan_photo_album()`。
- `scan_photo_album()` 会遍历目录、对每个媒体文件做路径解析和 `stat`，然后在全局 `photo_index_lock` 内读写同一个 `index.json`，并用 `json.dump(..., indent=2)` 重写全量索引。

这意味着现有索引只减少了部分元数据重新计算，并没有省掉最贵的目录遍历、全量索引读写和锁等待。随着相册数量、媒体数量和 NAS 路径增加，启动、刷新、滚动加载下一页、切换排序都会变慢。大照片质量高主要影响首次缩略图生成，不是目录扫描和排序变慢的主因。

## Goals / Non-Goals

**Goals:**

- 应用启动不再等待本地图库相册扫描，默认模型预览体验不被图库规模拖慢。
- 相册列表使用缓存摘要作为主路径，不因普通列表请求扫描所有相册。
- 相册分页、筛选和排序使用缓存索引作为主路径，不因普通浏览请求重新扫描目录。
- 将全局单文件索引演进为按相册拆分的索引，并保留媒体 ID 到相册的轻量查找能力。
- 保留现有媒体 ID、路径安全校验、图片/视频预览、下载、缩略图、视频 poster 和照片转 3D 行为。
- 明确扫描时机：首次无缓存、用户显式刷新、后台维护可以扫描；普通读取允许短暂使用过期缓存。
- 控制缓存增长风险，提供基础缓存统计/清理能力或为后续缓存管理入口留下稳定 API 边界。
- 缩略图和 video poster 继续按需生成，优先保证可见媒体体验，不用全量预热换取速度。

**Non-Goals:**

- 不实现视频转码、代理视频、HLS/DASH 或媒体重编码缓存。
- 不缓存原始图片或原始视频文件。
- 不默认为所有相册全量生成缩略图或视频封面。
- 不在第一阶段重写瀑布流虚拟化；虚拟化作为后续独立优化，避免破坏当前 Apple 风布局和滚动体验。
- 不修改 `ml-sharp/`、legacy `templates/index.html` 或 `static/lib/`。
- 不引入数据库、ORM、Tailwind、CSS-in-JS、axios 或新的重型前端状态库。

## Decisions

### Decision 1: 启动流程与图库相册加载解耦

**Why:** 默认 `activeView` 是模型页。启动阶段等待相册扫描会让完全不使用图库的路径也变慢，且扫描耗时会随着相册规模线性增长。启动时只需要认证、设置、任务和模型列表能支撑主界面即可。

具体方向：

- `App.loadPrivateData()` 不把 `fetchPhotoAlbums()` 作为 boot 阻塞项。
- 进入图库视图或侧栏需要展示相册时再加载相册列表。
- 可以在启动后空闲时后台拉取轻量相册摘要，但不能阻塞 `setBootComplete()`。

**Alternatives considered:**

- 保持启动时加载全部私有数据：实现简单，但继续让图库规模拖慢整个项目。
- 启动时并行请求所有数据：能减少串行等待，但只要相册扫描仍在 boot 前完成，最慢请求仍会决定启动体验。
- 启动时完全不加载模型列表：对默认模型页体验反而变差，不符合当前问题重心。

### Decision 2: 普通相册列表、分页、排序和筛选必须缓存优先短路

**Why:** 现有缓存已经存在，但普通请求仍进入 `scan_photo_album()`，导致每次滚动加载下一页和切排序都重新遍历目录。真正的优化是让正常读取路径根本不扫描目录。

具体方向：

- `/api/photo-albums` 返回相册配置和缓存摘要；无缓存时返回待扫描/未知数量状态，而不是阻塞扫描。
- `/api/photo-albums/<album_id>/photos` 从相册索引读取、筛选、排序、分页并返回结果。
- 普通请求允许短暂过期缓存；单文件预览、下载、缩略图仍做路径存在性和 root 校验。
- 用户点击刷新或显式扫描接口才强制扫描目录并更新索引。

**Alternatives considered:**

- 每次请求先做目录新鲜度校验：看似保证准确，但递归校验本身接近扫描，会破坏“既要启动快又要相册快”的目标。
- 前端只做懒加载分页：无法解决后端每页都扫描和重写索引的问题。
- 增加更大的页大小减少请求次数：会推迟但不消除扫描和 DOM 压力。

### Decision 3: 拆分索引为 catalog + 每相册索引，并使用可解析的 media_id 取代全局查找表

**Why:** 全局 `index.json` 会让所有相册共享一个大 JSON 和一个全局锁，每次扫描或更新都要读写全量索引。仅仅"拆成每相册索引"并不够：媒体解析接口（预览、下载、缩略图、video poster、照片转 3D）只拿到 `media_id`，而当前 `media_id = sha256(album_id \0 relative_path)[:32]` 是**单向哈希、无法反推 album_id**，所以会被迫再引入一个全局 `media-lookup.json` 映射表。

但全局 lookup 表会**把刚被消灭的"全局单文件 + 全局写锁"瓶颈原样请回来**：任意相册 rescan 都会增删该相册的 media_id，从而必须重写整张全局 lookup。

```text
            全局 lookup 表方案（被否决）            可解析 media_id 方案（采用）
        ┌──────────────────────────┐        ┌──────────────────────────┐
        │ catalog.json        (小) │        │ catalog.json        (小) │
        │ albums/<id>.json (每相册)│        │ albums/<id>.json (每相册)│
        │ media-lookup.json ◀──────┼─ 全量! │ (无 lookup 文件)         │
        └──────────────────────────┘        └──────────────────────────┘
          任意相册扫描 → 重写全局              任意相册扫描 → 只写
          lookup + 抢全局写锁                  catalog + 该相册索引
```

因此采用**可解析 media_id**：把 `album_id` 直接编码进 `media_id`，让解析阶段无需任何全局表即可定位相册。

新 media_id 方案：

```text
media_id = "<album_id>" + "_" + sha256(relative_path)[:N]
            └ 16 hex ──┘         └─ 相册内定位 ──┘

resolve_media_path(media_id):
    album_id = media_id.split("_", 1)[0]      # 直接切出 album_id，零查表
    album    = find_photo_album(album_id)     # 未配置/已删除 → not found
    index    = load_album_index(album_id)     # 只读 albums/<album_id>.json
    meta     = index["media"][media_id]       # 相册索引本就以 media_id 为 key
    # 仍必须做 root 包含校验 + 文件存在性校验后才返回内容
```

建议缓存结构：

```text
.photo-gallery-cache/
├── catalog.json                # 相册摘要：计数、封面 media_id、上次扫描时间、扫描状态、索引版本
├── albums/
│   ├── <album_id>.json         # 单相册媒体索引（以 media_id 为 key），含 source_version 与已缓存元数据
│   └── <album_id>.json
├── thumbnails/                 # 体积主要来源
└── video-posters/
```

- `catalog.json` 保存相册级摘要、计数、封面 `media_id`、上次扫描时间、扫描状态、索引版本号。
- `albums/<album_id>.json` 以 `media_id` 为 key 保存该相册媒体元数据（`relative_path / media_type / mtime / ctime / size / width / height` 及视频的 `duration / codec / bitrate`），并带相册级 `index_version`。
- 所有写入使用临时文件 + `os.replace` 原子替换；内容未变（按 `source_version` 比对）不写盘。
- 不再需要 `media-lookup.json`：解析路径完全由 `media_id` 内嵌的 `album_id` 驱动，正常读写路径上**没有任何全局可变文件**。

**Trade-off（已与用户确认接受）:** `media_id` 格式从纯哈希改为 `<album_id>_<hash>`，会使所有以旧 `media_id` 命名的缩略图/video poster 缓存文件失效，迁移后需按需重新生成一遍。当前项目尚未发版、缓存规模可控、用户为单人自用，一次性缩略图重算成本可接受，换取长期没有全局写瓶颈的干净结构。注意 `media_id` 改格式**不影响首扫复用旧元数据**——复用按 `(album_id, relative_path)` 匹配而非按 id，详见 Decision 5 与 Migration Plan。

**Alternatives considered:**

- 继续使用全局 `index.json`：改动少，但无法解决全量读写和全局锁瓶颈。
- 每相册索引 + 全局 `media-lookup.json`：分页快了，但 lookup 表本身又是"全局单文件 + 全局写锁"，任意相册 rescan 都要重写全量映射，等于把瓶颈改名保留；且缩略图缓存不失效。被否决，因为长期结构不干净。
- 每相册索引但解析时遍历所有相册索引：免去 lookup 文件，但 `resolve_media_path(media_id)` 要打开所有相册索引逐个查找，把高频的预览/下载路径变慢。
- 引入 SQLite：索引查询能力强，但项目目前明确无数据库架构，引入数据库和迁移成本过高。

### Decision 4: 增加进程内索引缓存和相册级锁

**Why:** 拆分 JSON 后，如果每次分页都从磁盘读相册索引，仍会产生不必要 IO。进程内缓存可以让筛选、排序、分页在内存完成；相册级锁可以避免一个相册扫描阻塞其它相册浏览。

具体方向：

- 维护按 `album_id` 的内存索引缓存，基于索引文件 `mtime` 或 `index_version` 判断是否复用。
- 维护相册级锁，扫描 A 相册不阻塞 B 相册分页。
- 单独保护 `catalog.json` 的短写入，不让长时间目录扫描持有全局锁。
- 排序第一版使用内存排序；只有观察到大规模排序仍慢时再缓存 `sort_key -> sorted_ids`。
- **读时快照语义**：分页/筛选/排序请求在进入时获取当前相册索引的一个不可变快照（引用或浅拷贝），后续后台扫描完成后只更新缓存中的"下一个版本"，不就地改写正在被某次浏览迭代的列表。这样后台刷新不会让正在滚动的会话发生跳条、重复或排序跳动。

**关于分页游标与后台刷新一致性：** 当前分页使用整数 offset 游标（`items[cursor:cursor+limit]`）。若后台扫描在翻页途中改变了列表长度或顺序，offset 游标会错位（漏图或重复）。配合上面的读时快照语义，约定：**同一次连续浏览/滚动使用同一索引快照版本，后台刷新得到的新版本在下次进入相册或显式刷新时才生效**，避免在用户滚动过程中重排其正在查看的列表。

**Alternatives considered:**

- 只依赖操作系统文件缓存：简单但不稳定，不能避免 JSON parse 和锁竞争。
- 全局锁保留不变：实现简单，但扫描/写入期间所有图库操作继续串行。
- 第一版即实现复杂排序索引：可能过度工程化；当前最大瓶颈不是纯内存排序。
- 后台扫描就地改写当前浏览列表：实现直接，但会在用户滚动时引发跳条/重排，违背"进相册快且稳"的目标。

### Decision 5: 扫描从阻塞读取改为显式/后台维护

**Why:** 用户更在乎打开和浏览是否立即可用。文件系统变化可以允许短暂延迟反映，但不应让普通读取每次都付出扫描成本。

具体方向：

- 首次无索引时，进入相册可以触发扫描；UI 显示扫描中/建立索引中。
- 有索引时，先返回缓存第一页；后台扫描可更新 catalog 和索引，但不打断当前滚动。
- 用户点击刷新时强制扫描当前相册，并在完成后刷新列表和计数。
- 后台扫描应低并发、可取消或可忽略过期结果，避免和缩略图生成、视频播放、3D 预览争资源。

**首次 catalog 填充（bootstrap）：** 启动不扫描会带来一个鸡生蛋问题——全新部署或迁移前从未扫过时，`catalog.json` 为空，切到图库会看到一整屏"待索引"占位（无封面、无计数）。约定：

- **老用户（存在旧 `index.json`）**：升级后首次需要相册列表时，从旧索引**轻量折算出 `catalog.json`**（仅按 album_id 分组统计计数、挑选封面、记录时间，不走文件系统、不 ffprobe），做到列表封面/计数立即可用；各相册的 `albums/<album_id>.json` 媒体索引则在首次进入相册或后台 bootstrap 时由"首扫复用旧元数据"建立（见 Migration Plan 方案 C）。
- **全新相册（无任何缓存）**：相册仍立即可见并显示"待索引"状态；进入图库视图后，由**低优先级后台任务逐个相册建立摘要**，封面/计数陆续浮现。后台 bootstrap 必须可被前台操作（用户主动进入某相册、显式刷新）打断或让路，且不阻塞 boot 与模型预览。

**Alternatives considered:**

- 所有扫描同步完成后才返回列表：数据最准确，但体验最差。
- 完全不后台刷新，只依赖手动刷新：实现简单，但相册外部变化反馈弱。
- 文件系统监听：实时性好，但跨平台/NAS 可靠性和复杂度不适合第一阶段。
- 全新相册不做任何 bootstrap、纯等用户逐个点开：实现最省，但首屏是一整屏空占位，体验比现状还差，故仅作为降级兜底而非默认。

### Decision 6: 缩略图和 video poster 保持按需生成，并加限流和可见优先

**Why:** 大照片质量高主要拖慢首次缩略图生成。直接全量预热几千张相机照片会导致 CPU、磁盘和缓存体积暴涨。更好的策略是可见优先、小范围预取、限并发。

具体方向：

- 缩略图/poster 缓存仍由源文件 `mtime + size` 绑定版本。
- 可见媒体请求优先生成；离屏媒体不阻塞列表。
- 后端缩略图生成使用并发限制，避免多个大图同时解码压垮设备。
- 可选后台预热只处理当前相册可见附近的少量媒体，不全量扫生成。

**Alternatives considered:**

- 全量预生成缩略图：再次打开很快，但首次成本和磁盘膨胀不可控。
- 完全不预取：实现简单，但快速滚动时占位更多。
- 前端用原图当缩略图：大照片会造成网络、解码和内存压力，违背大相册浏览目标。

### Decision 7: 缓存管理先做基础能力，UI 入口按范围控制

**Why:** 索引拆分本身不会显著增加体积；真正大头是 `thumbnails/` 和 `video-posters/`。已有删除相册清理对应缓存和临时 ZIP TTL 清理，后续需要让 owner 能理解和控制缓存。

具体方向：

- 提供缓存统计能力：索引、缩略图、poster、临时 ZIP 的大小和数量。
- 提供 owner-only 清理能力，至少支持清理失效缓存或清理全部图库生成缓存。
- 删除相册继续清理对应媒体 ID 的缩略图和 poster。
- 不删除用户原始照片或视频。

**Alternatives considered:**

- 本次不做缓存管理：主线更短，但用户已经关注 `.photo-gallery-cache` 增长，缺少治理入口。
- 自动按固定上限删除缩略图：可能导致用户刚浏览过的相册缩略图被误删，需更细的 LRU 设计，第一版不强推。

## Risks / Trade-offs

- [缓存短暂过期导致数量或封面不完全准确] -> 普通浏览使用缓存，显式刷新和后台扫描更新；UI 显示扫描状态或上次更新时间，单文件访问仍做存在性校验。
- [索引结构迁移影响媒体 ID 解析] -> 改用可解析 media_id（内嵌 album_id），解析无需全局表；首扫复用旧 index 后写 catalog + 每相册索引，测试覆盖预览、下载、缩略图、poster、照片转 3D、删除相册。
- [media_id 格式变更使旧缩略图/poster 缓存失效] -> 已与用户确认接受：迁移后按需重新生成缩略图/poster；规模可控、尚未发版，一次性成本换取长期无全局写瓶颈。
- [首扫不复用历史元数据触发 ffprobe 风暴] -> 首扫把旧 `index.json` 作为按 `(album_id, relative_path)` 的历史元数据来源复用 `width/height` 与视频 `duration/codec/bitrate`，mtime+size 命中即跳过 ffprobe；旧索引损坏才降级重建；测试断言首扫复用路径零 ffprobe 调用。
- [拆分索引后并发写入产生不一致] -> 使用相册级锁、短生命周期 catalog 写锁、临时文件 + 原子替换；扫描完成后统一提交。
- [后台扫描/刷新在滚动中改变列表导致跳条] -> 读时快照语义，同一浏览会话锁定一份索引版本，后台新版本在下次进入或显式刷新时生效。
- [后台扫描抢占 CPU/IO] -> 限制扫描并发，避免启动时全量扫描；后台扫描与 bootstrap 不能阻塞 boot 和普通分页，且可被前台操作让路。
- [缩略图生成仍可能拖慢大照片首次浏览] -> 限并发、可见优先、保留占位；不全量生成。
- [缓存体积继续增长] -> 删除相册清理对应缓存，保留临时 ZIP TTL 清理，新增缓存统计/owner 清理能力。
- [旧 `index.json` 与新结构并存造成困惑] -> 迁移后写入新结构，旧 `index.json` 仅作为迁移源，迁移完成后可归档/移除；文档和缓存统计明确新缓存布局。
- [瀑布流 DOM 在极深滚动后仍变重] -> 本次先解决后端扫描和索引瓶颈；如仍明显卡顿，再单独探索瀑布流虚拟化并做视觉回归。

## Migration Plan

迁移是**运行时自动、对每个 workspace 一次性生效**的行为，不是开发者侧脚本，也不是常驻迁移程序（自部署场景下开发者接触不到用户 workspace）。为控制复杂度，**不写专门的 migration transform**，而是让"首次扫描"顺手复用旧索引（方案 C），损坏时降级为纯重建（方案 B 兜底）。

1. 增加新缓存路径与索引读写模块（`catalog.json`、`albums/<album_id>.json`），支持原子写入、内容未变不写盘和 JSON 读取容错。
2. 引入可解析 media_id（`<album_id>_<hash>`）与 `resolve_media_path` 的零查表解析；让缩略图、poster、下载、预览和照片转 3D 走新解析路径。
3. **首扫复用旧 `index.json`（方案 C，主路径）**：当某相册尚无 `albums/<album_id>.json` 时，首次扫描把旧全局 `index.json` 作为"历史元数据来源"，按 `(album_id, relative_path)` 匹配复用已缓存的图片尺寸与视频 `duration/codec/bitrate`。由于复用按相对路径而非 media_id 匹配，media_id 改格式不影响复用，且 mtime+size 命中即跳过 ffprobe。首扫仍需一次 `os.walk` 建立每相册索引，期间 UI 显示"建立索引中"。
4. **损坏/缺失时降级重建（方案 B，兜底）**：旧 `index.json` 不存在或无法解析时，直接全新扫描重建；此时视频会重新 ffprobe，UI 同样显示"建立索引中"提示。
5. 首扫成功写出新结构后，把旧 `index.json` 改名归档（如 `index.json.bak`），之后不再读取；旧缩略图/poster 因 media_id 改格式而失效，按需以新 media_id 重新生成。
6. 将相册列表改为 catalog 摘要主路径，无缓存时返回待扫描状态，不阻塞扫描所有相册。
7. 将相册分页、筛选、排序改为读取单相册索引（读时快照）；显式刷新接口负责扫描并更新索引与 catalog。
8. 调整前端启动流程，使图库相册加载不再阻塞 boot；进入图库视图后加载相册摘要，全新相册由低优先级后台 bootstrap 填充摘要。
9. 增加扫描状态、刷新反馈、缓存统计/清理入口与对应 i18n 文案。
10. 增加测试覆盖首扫复用旧索引（含视频元数据零 ffprobe 断言）、损坏降级重建、缓存优先读取、显式扫描、media_id 解析、权限、缓存清理和旧流程回归。

**Rollback:** 如果新索引路径出现问题，可以回退到旧 `scan_photo_album` 主路径和旧 `index.json` 读取（保留旧文件直到迁移验证通过再归档）。旧相册配置、原始媒体文件不应被删除；缩略图/poster 即便被清理也可重新生成。验证点包括应用启动可进入模型页、相册列表可打开、旧图片/视频可预览和下载、照片转 3D 可排队、删除相册不删除原始文件。

## Success Metrics

迁移后应满足以下可验证指标（对应 tasks 中的性能验证）：

- **启动与图库规模解耦**：boot 完成时间不随相册数量或媒体数量增长（O(1)），启动路径不触发任何相册目录 `os.walk`。
- **暖索引下相册第一页快速可用**：已建立索引的相册，打开后第一页 TTFB 处于交互级（目标 < ~300–500ms，取决于设备/磁盘）。
- **普通浏览零目录扫描**：翻页、切排序、切筛选在有可用索引时**零次** `os.walk`（可用日志埋点或测试断言验证）。
- **首扫复用无损**：首次浏览旧相册（存在可用旧 `index.json`）时，复用历史元数据，视频零 ffprobe 调用、图片不重新读取尺寸。
- **跨相册不互相阻塞**：扫描一个大/NAS 相册期间，其它已索引相册仍可正常分页。

## Open Questions

- 是否在第一版前端加入"清理全部图库缓存"按钮，还是只提供 owner-only API 和文档入口。
- 后台扫描/bootstrap 是否需要展示进度百分比，还是只展示"扫描中/已完成/失败"状态。
- 缩略图预热是否需要用户可配置开关，默认是否只做可见附近少量预取。
- 后续是否单独开 change 做瀑布流虚拟化，覆盖数万媒体条目的极端滚动场景。
- 可解析 media_id 的分隔符与长度（`<album_id>_<hash>` 的 hash 截断位数）取多少，既避免单相册内碰撞又控制 id 长度。
