## Context

Sharp GUI 当前的本地照片图库已经具备相册目录配置、路径安全校验、分页浏览、照片缩略图、原图预览、下载、批量选择和照片转 3D 能力。用户提供的界面截图显示，现有照片图库已经形成清晰的 Apple 风格：浅色毛玻璃侧边栏、顶部悬浮工具条、瀑布流卡片、胶囊按钮、圆形图标按钮和全屏沉浸式预览。

视频支持会跨越后端索引、缓存、API、权限、前端类型、状态、网格、预览器、国际化和响应式 UI。它同时涉及浏览器格式支持、Range 请求、视频元数据、封面抽帧和高码率文件播放体验，因此需要先确定技术边界，避免把图库预览演变成复杂的视频平台。

## Goals / Non-Goals

**Goals:**

- 将本地照片图库自然扩展为本地媒体图库，支持图片和视频混合浏览。
- 在图库界面提供全部、照片、视频三种类型筛选。
- 为视频卡片提供封面、播放标识、时长、分辨率和下载入口，并与现有瀑布流视觉保持一致。
- 提供美观的自定义视频预览界面，支持播放、暂停、进度拖动、移动端长按画面精细 seek、音量/静音、全屏、下载、关闭和前后媒体切换。
- 优先使用浏览器原生视频解码和系统硬件加速能力，减少前端体积和播放风险。
- 对 PC 与移动端做明确适配，保证鼠标、键盘、触控、窄屏布局都可用。
- 对无法播放的视频提供清晰状态，提示用户下载到本地播放器打开。

**Non-Goals:**

- 不实现视频转 3D。
- 不实现视频剪辑、裁切、调色、字幕编辑或转码队列管理界面。
- 不生成兼容代理视频，不做自动转码。
- 不默认构建多码率 HLS/DASH 流媒体平台。
- 不实现 DRM、加密视频播放或云端同步。
- 不处理 HDR 色彩管理；本次只覆盖普通 SDR 视频体验。
- 不修改 `ml-sharp/`、legacy `templates/index.html` 或 `static/lib/`。

## Decisions

### Decision 1: 使用原生 `<video>` 作为播放内核，自定义 React 控制层

**Why:** 浏览器原生 `<video>` 可以直接利用浏览器和操作系统的视频解码、硬件加速、缓冲、seek、全屏和音频能力。Sharp GUI 的目标是本地/NAS 媒体预览，不是自建完整视频播放平台；将解码交给浏览器，项目只负责视觉、状态、交互和降级提示，可以获得最好的复杂度收益比。

**Alternatives considered:**

- Video.js：成熟但体积和样式体系较重，容易带来框架自带播放器观感；本次只需要本地文件预览，不值得引入。
- Shaka Player：适合 DASH/HLS/ABR/DRM，但对本地相册直出预览过重。
- hls.js：适合 HLS，不解决普通本地 MP4/MOV/WebM 文件直出预览问题。
- WebCodecs/wasm FFmpeg 自研解码：控制力强但代价极高，会引入音画同步、性能、功耗、移动端兼容和内存风险。

### Decision 2: 将前端概念从 PhotoItem 扩展为 MediaItem，而不是为视频复制一套图库

**Why:** 现有相册配置、分页、瀑布流、选择、下载、空状态和刷新行为大多可以复用。用 `media_type: "image" | "video"` 扩展条目，比维护两个平行图库更容易保持 UI 和权限一致。

**Alternatives considered:**

- 新建独立视频图库入口：边界清晰，但用户需要在照片和视频之间来回切换，不符合“相册目录混合媒体”的真实使用方式。
- 在当前 PhotoItem 上临时塞视频字段：改动少但类型语义混乱，后续筛选、预览、下载和测试会变脆。

### Decision 3: 后端保留稳定 ID 和 root 校验，新增媒体索引字段

**Why:** 现有 photo id 通过 album id + relative path 生成，不暴露绝对路径，并且每次解析都校验文件仍在相册 root 内。这是视频预览和下载必须继承的安全基础。

新增索引字段应包括：`media_type`、`mime_type`、`duration`、`width`、`height`、`poster_url`、`preview_url`、`download_url`、`playback_url`、`video_codec`、`audio_codec`、`bitrate`、`playback_status` 等。字段可以按阶段渐进填充，列表接口必须允许部分字段为空。

**Alternatives considered:**

- 直接按文件路径请求视频：实现简单但会暴露路径并增加路径遍历风险。
- 独立视频索引文件：可隔离历史索引，但会增加相册扫描和分页合并复杂度。

### Decision 4: 视频元数据与封面采用可选 FFmpeg/ffprobe 增强

**Why:** `ffprobe` 用于读取视频宽高、时长、编码和码率；`ffmpeg` 用于从视频中抽取一帧生成图库封面。Sharp GUI 已经有本地安装/便携包场景，FFmpeg 可以作为可检测的增强能力：可用时生成准确元数据和封面，不可用时仍可列出视频并让浏览器尝试播放。

视频文件“可能”带有内嵌封面，但不能作为图库封面的稳定来源：很多手机或相机视频没有内嵌封面；不同容器和工具对封面流的写法不一致；浏览器 `<video>` 不提供可靠的批量读取封面 API；直接在瀑布流里创建大量隐藏 video 元素去取首帧会触发大量网络读取、解码和内存占用。后端按需抽一帧并缓存，能让图库列表更稳定、更轻，也能统一封面尺寸和失效策略。

**Alternatives considered:**

- 完全不用 FFmpeg：依赖前端 `<video>` loadedmetadata 获取元数据，后端列表无法提前展示时长和稳定封面，大相册体验会下降。
- 强制安装 FFmpeg：体验更完整，但会增加安装包、平台适配和首次配置门槛。

### Decision 5: 视频只做原文件直出播放，失败后提示下载

**Why:** 对 MP4/MOV/WebM 等浏览器可能直接支持的 SDR 视频，通过 `/api/media-original/<id>` 或类似端点直出原文件，并确保响应支持 Range。若浏览器无法加载或解码，预览界面直接显示“当前浏览器无法播放此视频，请下载到本地播放”的本地化提示，同时保留下载和关闭操作。

这样可以避免转码、代理文件缓存、转码进度、磁盘占用和 CPU/GPU 消耗，把第一阶段控制在“可靠浏览与预览”范围内。

**Alternatives considered:**

- 所有视频预转码：播放兼容性更好，但 CPU/GPU、磁盘、等待时间和失败率都高，超出本次目标。
- 生成兼容 proxy MP4：可以解决部分浏览器不支持问题，但会引入转码生命周期和缓存清理复杂度。
- 默认生成 HLS：对高规格视频和移动网络有潜力，但多码率切片复杂，第一阶段不需要。

### Decision 6: 画面精细拖动 seek 仅支持移动端

**Why:** 移动端没有鼠标精确指针，视频画面上的长按横向拖动可以补足小屏进度条不易精细操作的问题。PC 端保留进度条拖动和键盘操作即可，避免画面拖动和点击播放/暂停、文本选择、浏览器手势产生冲突。

建议交互：

- PC 端：单击画面播放/暂停，进度条拖动 seek，不启用画面拖动 seek。
- 移动端：长按视频画面后横向拖动进入精细 seek，显示浮层时间偏移和目标时间。
- 移动端拖动结束后，视频 seek 到目标位置，并按拖动前状态恢复播放或保持暂停。

**Alternatives considered:**

- PC 和移动端都支持画面拖动：功能一致，但 PC 端误触和交互冲突更多，收益不大。
- 只用进度条拖动：实现最简单，但移动端小屏上精细定位体验较弱。
- 画面拖动映射大范围进度：容易误触且不精细，移动端体验差。

### Decision 7: UI 使用现有 Apple 毛玻璃语言，不引入外部 UI 框架

**Why:** 用户最关注最终效果和界面美观，且现有截图已经有明确风格。播放器控件应使用 CSS Modules、CSS Variables、lucide/现有图标、圆形玻璃按钮、胶囊底栏和沉浸式模糊背景。避免引入 UI 框架导致视觉不一致。

**Alternatives considered:**

- 使用播放器默认 controls：实现最快，但无法匹配现有高机位 Apple 风界面。
- 使用现成播放器皮肤：可快速上线，但精细手势、移动端布局和视觉统一会受限。

## Risks / Trade-offs

- [浏览器不支持某些容器或编码] -> 预览失败时显示“请下载到本地播放”的本地化提示，并保留下载按钮。
- [高码率/高分辨率视频播放卡顿] -> 保持原文件直出 + Range；避免在网格中加载真实视频；播放失败或卡顿时用户仍可下载原文件。
- [FFmpeg 不存在或执行失败] -> 将 FFmpeg 设为增强能力；失败时保留视频条目和原文件播放尝试，封面使用占位状态。
- [抽帧消耗资源] -> 缓存封面；绑定源文件 mtime/size 失效；限制并发；避免列表请求同步长时间阻塞。
- [移动端误触画面拖动 seek] -> 长按或明确拖动阈值后进入 scrub；拖动时显示浮层反馈；单击仍只处理播放/暂停。
- [下载大视频占用内存] -> 下载视频使用浏览器直接打开下载 URL，不通过前端 fetch 成 blob；批量下载仍需限制数量和总大小。
- [照片转 3D 行为被视频选择污染] -> 转 3D 操作只对图片可用；选择栏根据选中媒体类型显示可用操作。
- [命名从 Photo 到 Media 改动较大] -> 优先保持旧 API 兼容，必要时新增媒体端点；前端逐步迁移类型，避免一次性破坏照片流程。

## Migration Plan

1. 扩展后端媒体识别与索引，保持旧照片字段和旧接口可用。
2. 新增或扩展列表 API 支持 `type=all|photo|video`，默认 `all` 或按兼容策略返回现有照片结果。
3. 新增视频封面、播放和下载端点，并纳入权限矩阵。
4. 前端类型从照片条目扩展到媒体条目，保留照片转 3D 逻辑只作用于图片。
5. 将 `ImageViewer` 演进为媒体预览器或新增 `MediaViewer`，图片路径复用现有行为，视频路径启用自定义播放器。
6. 加入 UI 设计图对齐后的样式与响应式布局。
7. 增加测试覆盖后再切换默认图库筛选和 UI 文案。

**Rollback:** 如果视频能力出现问题，可关闭视频扩展名扫描或隐藏视频筛选入口；旧照片索引、照片列表、缩略图、预览和转 3D 行为应保持可回退。验证点包括旧照片相册仍可打开、照片缩略图仍可生成、原图预览和下载仍可用、照片转 3D 仍只接受图片。

## Open Questions

- FFmpeg/ffprobe 是否纳入安装脚本和便携包，还是仅做运行时检测并提示用户自行安装。
- 移动端画面拖动 seek 的默认灵敏度应以秒/像素还是按视频时长比例计算。
- 是否需要为超大视频批量下载设置总大小上限。
